Return Recovery Dashboard

User Journey — Returns Receiving SLA Breach

End-to-end flow from alert discovery to physical search, triage, and loop closure. Persona: Warehouse Manager at Mumbai FC-1.

Phase 1 — Entry
User Action
Step 01
Warehouse Manager opens the Return Recovery Dashboard in UniReco
The Warehouse Manager logs into UniReco at the start of their shift. The Return Recovery Dashboard opens directly — scoped to their facility. No link or attachment needed.
🏢
Facility scope: Dashboard is pre-filtered to Mumbai FC-1 — all alerts, widgets, and data are specific to this facility
📊
Copilot panel is visible on the right side of the screen with active alerts for the facility
Phase 2 — Alert Discovery
User Action
Step 02
Click the alert — “Returns Receiving SLA Breach · ₹0.82L”
The alert appears in the Copilot panel. The card shows the title, value, and subtext: 38 returns breached receiving SLA. HQ Notified. Close on Priority.
🔴
Returns Receiving SLA Breach · ₹0.82L  SLA Breached
🚫
HQ has been notified — there is external pressure to resolve this quickly
Phase 3 — Context & Stakes
System Surface
Step 03
Flyout opens with three structured sections
The detail panel loads in Warehouse View with the full context needed to act on the breach.
⚠️
What’s at stake: ₹0.82L in untracked returns at Mumbai FC-1 exposed to pilferage risk · HQ has been notified and may reach out
📋
What triggered this: 38 return shipments were delivered to Mumbai FC-1 but no manifest or putaway was created within the receiving SLA window
📊
Shipment table: 38 rows sorted by pending days descending · Order ID / AWB · Courier · Platform · Delivered date · Pending days (colour-coded) · Value · 15 rows per page
Phase 4 — Assign to Floor
User Action
Step 04
Print List or Share via WhatsApp to assign the locate task
The manager distributes the shipment list to warehouse staff before the physical search begins.
🖨
Print List: Generates a print-ready locate sheet — facility name, date, full shipment table with Order IDs and AWBs, and a worker signature line for physical handoff
📲
Share via WhatsApp: Sends the shipment list to the assigned person — auto-composed message with count, value, and a numbered list of all pending Order IDs and AWBs
Phase 5 — Set Check-back Timer
User Action
Step 05
Set a check-back timer so the alert reopens automatically after the search period
Once the locate list has been assigned, the manager sets a timer. When it expires, the alert is reinstated at the top of the Copilot panel with refreshed data.
⏱
Timer options: 30 min · 1 hr · 2 hrs — a live countdown is visible in the flyout once set
🔁
On expiry: Alert is reinstated at the top of the Copilot panel — only shipments not yet inwarded are shown when reopened
Phase 6 — Physical Search
Off-System Action
Step 06
Assigned person(s) search for shipments and inward found ones in WMS
Floor workers use the printed sheet or WhatsApp list to physically locate each shipment. Any shipment that is found is scanned and inwarded in WMS during this period.
📦
Found: Scanned and inwarded in WMS — exits the breach list once received
🚫
Not found: Remains in the shipment table — will be actioned by the manager after the timer ends
Phase 7 — Triage Remaining Shipments
User Action
Step 07
After the timer ends — mark shipments that could not be found as Missing
The flyout resurfaces with refreshed data showing only uninwarded shipments. The manager triages each remaining row.
⚠️
Mark Missing: Flags a specific row — row strikes through, summary bar updates, Disputed Returns widget increments live
↻
Unmark: Reverses Missing status on any row if the shipment is subsequently found
🚫
Mark All Missing: Flags all remaining pending rows at once if the entire batch could not be located
Row → Missing Disputed widget → Updated live
Phase 8 — Close the Loop
Decision Point
Were any shipments marked Missing when closing?
✓ All Located
Clean Close
Step 08A
Alert closes — Copilot moves to the next priority alert
All shipments were found and inwarded. The SLA breach alert archives cleanly with no follow-up required.
Alert → Archived Value → Resolved
✗ Some Missing
SPF Triggered
Step 08B
New alert generated — “Notify HQ — X Missing Returns Need SPF Claim”
SLA breach alert closes. A new alert is immediately injected into the Copilot rail for the missing shipments.
SLA Alert → Closed SPF Alert → Injected
Phase 9 — SPF Escalation
User Action
Step 09
New alert includes SCM Head contact and a pre-drafted email for SPF claim initiation
The manager clicks the new SPF alert. The flyout opens with everything needed to escalate to HQ in a single action.
👤
SCM Head contact details surfaced directly — email, phone, and WhatsApp
✉️
Pre-drafted email: Lists all missing Order IDs and AWBs · Subject auto-composed with count and value · Ready to send for SPF claim initiation at HQ
End State
Complete
Breach resolved — all shipments located or escalated for SPF claim
Every breached shipment at Mumbai FC-1 has been triaged. Found shipments are inwarded in WMS. Missing shipments have been escalated to the SCM Head with Order IDs and AWBs for SPF claim initiation. Both alerts are closed on the Return Recovery Dashboard.
Found shipments → Inwarded in WMS Missing shipments → SPF claim initiated Both alerts → Closed
